## Start with the human version

SPECTER is not a chain.

It is not a token.

It is not a new wallet standard you have to memorize before using it.

SPECTER is a **private receiving system**.

You publish one public profile, and each sender turns that profile into a different one-time receiving address.

To everyone else, the payment lands at an address that looks unrelated to you.

## If you are new to this space

### What is a stealth address?

A stealth address is a one-time destination created for a single payment.

You do not reuse it.

You do not hand it out in advance.

The sender derives it from your public **meta-address**, and you later recover it by scanning announcements.

### What is a meta-address?

A meta-address is SPECTER's public receiving profile.

It bundles:

- `spending_pk`
- `viewing_pk`

In the current backend, both come from ML-KEM key generation in `specter/specter-api/src/handlers.rs`.

### What is an announcement?

An announcement is the public breadcrumb that lets you find your payment later.

The current wire format includes:

- `ephemeral_key` as the ML-KEM ciphertext
- `view_tag` as a 1-byte scan filter
- optional `channel_id`, `tx_hash`, `amount`, and `chain`

That structure is defined in `specter/specter-core/src/types/announcement.rs`.

## Think of it like this

  <Step title="You publish a mailbox blueprint" icon="mail">
    Your meta-address is not a wallet you receive into directly. It is a public instruction set that lets someone create a one-time mailbox for you.
  </Step>
  <Step title="The sender creates a fresh mailbox" icon="sparkles">
    They use your public data to derive a new stealth address that has not appeared before.

  </Step>

    Your viewing secret key lets you find which announcement is yours and recover the spendable result.
  </Step>

## What does post-quantum mean here?

It means the privacy layer is not relying on classical elliptic-curve key exchange.

Instead, SPECTER uses **ML-KEM-768**, the finalized NIST standard for key encapsulation, implemented in `specter/specter-crypto/src/kyber.rs`.

That matters because stealth systems leave cryptographic breadcrumbs on-chain for a very long time.

If your discovery layer depended only on classical ECDH, a future attacker with a strong enough quantum computer could potentially use old public data to break the privacy of old payments.

SPECTER avoids that specific failure mode on the receiving side by using ML-KEM for the sender-to-recipient shared secret.

## What are the keys actually doing?

<AccordionGroup>
  <Accordion title="`viewing_pk` and `viewing_sk`">
    These are for **discovery**.

    The sender encapsulates to `viewing_pk`.

    You later decapsulate with `viewing_sk` while scanning announcements.
  </Accordion>
  <Accordion title="`spending_pk`">
    In the current wallet-compatible flow, the sender combines the ML-KEM shared secret with `spending_pk` to deterministically derive the one-time stealth address.

    That logic lives in `specter/specter-crypto/src/derive.rs`.
  </Accordion>
  <Accordion title="The spendable result">
    When you scan successfully, the API returns an `eth_private_key` derived from the same shared secret path.

    That is what makes the resulting address usable in existing Ethereum wallet flows today.
  </Accordion>
</AccordionGroup>

## The honest security summary

<Warning>
SPECTER today is best described as **post-quantum recipient privacy with classical spend security**.
</Warning>

Why:

- The sender-recipient shared secret uses ML-KEM-768.
- The current spendable wallet output is still a derived `secp256k1` key for Ethereum compatibility.

The full explanation is on **[Quantum safety boundaries](/core-concepts/quantum-safety-boundaries)**.
