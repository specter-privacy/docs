## Start with the right repo

SPECTER work currently spans two codebases:

| Repo | What belongs there |
|---|---|
| [Product repo](https://github.com/pranshurastogi/SPECTER) | Rust backend, frontend app, GitHub issues, CI, and product implementation work |
| This Mintlify docs repo | Public documentation structure, copy, charts, and source-backed explanation pages |

If your change affects product behavior, you will usually need updates in both places.

## Start with an issue

Use **[Contribution Opportunities](/build/contribution-opportunities)** to review the current GitHub backlog before you start coding.

If no issue matches your work:

1. Open or comment on a GitHub issue.
2. Describe the exact scope you plan to take.
3. Keep the PR focused on one problem or one tightly related set of changes.

## Contribution flow

<Steps>
  <Step title="Choose a scoped problem" icon="git-branch">
    Pick an existing GitHub issue or open a new one with a clear acceptance target.
  </Step>
  <Step title="Set up the workspace" icon="wrench">
    Follow **[Development Setup](/build/development-setup)** for the product repo and run the docs site locally if your change affects documentation.
  </Step>
  <Step title="Implement and verify" icon="code">
    Add tests, run local checks, and avoid mixing unrelated refactors into the same PR.
  </Step>
  <Step title="Update the docs" icon="book-open">
    If behavior changed, update the relevant guide, reference page, and **[Verification Matrix](/reference/verification-matrix)**.
  </Step>
  <Step title="Open the PR" icon="git-pull-request">
    Explain what changed, what you verified, and which issue it resolves.
  </Step>
</Steps>

## Quality bar

- **No unsafe code.** The Rust crates enforce `#![forbid(unsafe_code)]`.
- **Formatting.** Run `cargo fmt` before committing.
- **Linting.** Run `cargo clippy -- -D warnings`.
- **Tests.** New behavior needs tests. Bug fixes need regression coverage.
- **Documentation.** Document only verified behavior. Move aspirational items to **[Roadmap](/roadmap/yet-to-implement)**.
- **Small PRs.** Prefer reviewable changes over broad cleanup sweeps.

## What to verify locally

For product repo changes:

```bash
cargo fmt
cargo clippy -- -D warnings
cargo test
```

For docs repo changes:

```bash
mint dev
mint broken-links
```

If the Mintlify CLI is not installed yet, install it first in your local environment before running the docs checks.

## Documentation update rules

If you change a feature, update the docs in the same pass.

| Change type | Minimum docs follow-up |
|---|---|
| API route or payload change | Update the affected API page, quickstart/integration guide if needed, and the verification matrix |
| Security or status change | Update **Security Boundaries**, **FAQ**, and roadmap/status notes |
| Yellow integration behavior | Update **Yellow endpoints** and the related use-case page |
| Contributor workflow or repo layout | Update this page and **Contribution Opportunities** |

## Review criteria

Reviewers will look for:

- Correctness
- Security and secret-handling hygiene
- API compatibility
- Test coverage
- Clear docs for behavior changes
